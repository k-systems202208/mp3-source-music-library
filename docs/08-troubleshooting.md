# トラブルシューティング

## 1. 画面に「読み込み中」のまま

### 確認

1. 黒いサーバー画面が開いているか
2. ブラウザのURLが起動時に表示されたURLと一致するか
3. `/api/health`が開くか
4. F12のConsoleにJavaScriptエラーがないか

### 対応

- 黒い画面を閉じて再起動
- 別の古いタブを閉じる
- Ctrl+F5で再読込
- 同じポートを別プロセスが使用していないか確認

## 2. 「Failed to fetch」

主な原因:

- Pythonサーバーが停止
- PCがスリープ
- Wi-Fi切断
- Tailscale切断
- サーバー側例外
- URLのポート違い

### 切り分け

```text
/api/health
```

が開かない場合は通信・サーバー問題です。

開くが画面だけ失敗する場合は、F12のNetworkで`/api/browse`のHTTPステータスを確認します。

一時的な場合は再起動で復旧することがあります。再発時は黒い画面のエラー全文、操作手順、URLの`/api/browse`クエリを記録します。

## 3. 文字化け

### 対応順

1. 最新版で全MP3を再解析
2. `library-diagnostics.csv`を確認
3. MP3タグをタグ編集ソフトでUTF-16またはUTF-8へ保存し直す
4. 画面の補正機能を使用
5. 旧JSONが誤っている場合は移行データを見直す

補正機能はMP3タグを変更しません。

## 4. 再生ボタンが反応しない

### 確認

- 曲の`audioFile`パスが存在するか
- NetworkでMP3が200または206か
- 404になっていないか
- 日本語パスがURLエンコードされているか
- Audio要素にエラーが出ていないか

### 過去に発生した原因

曲IDが文字列なのにJavaScriptで数値化し、`NaN`になって再生対象を取得できない問題がありました。v2.4では文字列IDで統一済みです。

## 5. シークできない

MP3応答に次があるか確認します。

```text
206 Partial Content
Accept-Ranges: bytes
Content-Range
```

逆プロキシ等を追加した場合、Rangeヘッダーが転送されている必要があります。

## 6. 初回起動が遅い

正常です。次の場合は全曲再解析します。

- 初回
- DBスキーマ更新でsortタグ再取得が必要
- MP3の更新日時が変わった
- キャッシュ条件が満たされない

2回目以降の`SQLite cache hits`が多ければ正常です。

## 7. 2回目も遅い

- バックアップソフトがMP3更新日時を変更していないか
- NAS等でmtimeが安定しているか
- DBが毎回削除されていないか
- `schema_info`のbackfillフラグが保存されているか
- セキュリティソフトの全ファイル検査がないか

## 8. 曲が消えた

MP3が存在するのに消えた場合:

1. `Music`配下にあるか
2. 拡張子が`.mp3`か
3. スキャン時エラーがないか
4. `library-diagnostics.csv`確認
5. `stats`でavailable / unavailable確認

解析エラーでも既存曲は前回情報を保持する設計ですが、初回登録で読めないファイルは登録できない場合があります。

## 9. 同名曲が混ざる

曲の正本は相対パス別のMP3です。同名でも別曲IDになります。

アーティストやアルバムのタグが同一の場合、一覧グループとしてまとめられます。誤グループの場合はMP3タグを整理するか、将来の曲単位補正機能が必要です。

## 10. MP3移動後に再生回数が引き継がれない

次の場合、移動検出が安全のため成立しません。

- 内容が編集された
- 同じ署名候補が複数ある
- コピー元とコピー先が同時に存在する
- サイズ、先頭、末尾が変わった

手動DB編集は推奨しません。バックアップを取ってから対応します。

## 11. LANでPCからは開くがスマホから開かない

確認順:

1. サーバーが`0.0.0.0:8765`
2. PCとスマホが同じ通常Wi-Fi
3. ゲストWi-Fiではない
4. ファイアウォールルール有効
5. IPアドレスが変わっていない
6. VPNを一時的に停止
7. ルーターのAP isolationが無効

```powershell
Get-NetFirewallRule -DisplayName "Music Library LAN 8765*"
Get-NetTCPConnection -LocalPort 8765
```

## 12. NetworkCategoryをPrivateへ変更できない

エラーにGroup Policyが示される場合、端末管理ポリシーで禁止されています。

対処:

- 無理にレジストリ変更しない
- Public＋LocalSubnet4＋TCP 8765だけ許可
- 使用後にルールをDisable
- 公共Wi-Fiでは起動しない

## 13. TailscaleでPCが見えない

- 招待承認と同じアカウントか
- 正しいtailnetを選択しているか
- Connectedか
- UsersでActiveか
- Machinesで端末承認待ちではないか
- Access controlsで拒否されていないか

## 14. TailscaleでPCは見えるがサイトが開かない

PCで:

```powershell
tailscale serve status
```

ローカル確認:

```text
http://127.0.0.1:8765/api/health
```

Serveが空なら:

```powershell
tailscale serve --bg 8765
```

アクセス制御を絞った後は、招待ユーザーから`music-server:443`が許可されているか確認します。

## 15. 外出先で途中停止する

- 自宅PCがスリープしていないか
- ノートPCの蓋を閉じていないか
- TailscaleがConnectedか
- モバイル回線が不安定でないか
- iOS／Androidの省電力制御
- ブラウザのバックグラウンド再生制限

Webアプリはネイティブ音楽アプリよりバックグラウンド再生が不安定な場合があります。

## 16. 再生回数が想定より増える

複数端末の再生は合算されます。誰が再生したかは区別しません。

同じ曲を新規ロードし直す操作を繰り返すと、その都度加算されます。

## 17. 補正が別端末で戻ったように見える

別端末の古い表示が残っている可能性があります。再検索または再読込してください。

複数端末が同時更新した場合は最後の保存が有効です。

## 18. SQLite整合性

サーバー停止後:

```powershell
py -3 library-maintenance.py check
```

`ok`以外なら:

1. DBとWAL/SHMを退避
2. バックアップを復元
3. check
4. 再起動

## 19. ポート8765が使用中

```powershell
Get-NetTCPConnection -LocalPort 8765 -ErrorAction SilentlyContinue
```

該当プロセスを終了するか、別ポートへ変更します。BAT、ファイアウォール、URL、Tailscale Serveの転送先をすべて一致させます。

## 20. GitHubへ誤ってDBやMP3をpushした

`.gitignore`を追加するだけでは過去履歴から消えません。

1. リポジトリを一時非公開
2. Git履歴から対象を削除
3. 公開URL・招待情報があれば無効化
4. 新しいクリーンリポジトリで公開を検討
5. GitHubの機密情報削除手順に従う

音源を含めた場合は公開状態を直ちに解除します。
